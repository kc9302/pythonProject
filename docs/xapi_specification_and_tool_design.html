xAPI 규격 분석 및 데이터 기획 도구 설계 가이드

xAPI-Communication 및 Error Codes 명세 기반 인터랙티브 저작 도구 기능 기획

1. 개요 및 목적

본 문서는 ADL의 xAPI(Experience API) Specification(Communication 및 Error Codes)을 상세히 분석하고, 이를 바탕으로 데이터 기획자가 학습 데이터 모델을 직관적으로 설계할 수 있는 '웹 기반 xAPI 데이터 기획 및 검증 도구'의 기능적 설계 방향을 제시합니다.

데이터 기획자가 복잡한 JSON-LD 및 xAPI 스키마 규칙을 완벽히 숙지하지 않더라도, GUI 환경에서 드래그 앤 드롭 방식으로 Statement를 구성하고 실시간 유효성 검증(Validation) 피드백을 통해 완벽한 학습 데이터를 설계할 수 있도록 지원하는 시스템 개발을 목표로 합니다.

2. xAPI 통신 규격 및 에러 코드 분석 (LRS 명세)

LRS(Learning Record Store)와 학습용 에이전트 간의 데이터 교환 시, LRS는 전송된 Statement의 엄격한 유효성 검사를 수행하며 규칙 위반 시 HTTP 상태 코드를 통해 에러를 반환합니다. 설계할 기획 툴은 LRS 서버에 도달하기 전 전 단계(Client-side Validation)에서 이 규칙들을 사전에 완벽하게 검증해 주어야 합니다.

2.1 주요 HTTP 에러 코드 정의 및 툴 반영 전략

HTTP 상태 코드 LRS 발생 원인 및 규격 명세 기획 툴(기능) 구현 및 대응 전략
400 Bad Request - 필수 인자 누락 또는 유효하지 않은 인수
- 잘못 구성된 JSON 구조
- 스키마 요구사항 미준수 (데이터 타입 불일치 등)
- 대소문자 구분 오류 (Key의 CamelCase 위반) 		- 필수 필드 누락 시 실시간 붉은색 아웃라인 표시
- JSON 구조 자동 생성 (인간 실수 원천 차단)
- 포맷 규칙(UUID, IRI, Mailto) 정규식 실시간 검증
- 사전 정의된 올바른 Key 값만 바인딩
401 Unauthorized 인증이 필요하거나 제공된 자격 증명(Credential/API Key)이 거부된 경우. 툴 내 프로젝트 설정에서 LRS Endpoint 및 Basic Auth / OAuth 인증 정보 입력 및 테스트 연결 기능 제공.
403 Forbidden 자격 증명은 유효하나, 해당 토큰에 할당된 권한 권한(Scope)이 부족한 경우 (예: Read 전용 토큰으로 Write 시도). 기획 단계에서 권한 프로필(상태)을 시뮬레이션하여 권한 부족 시나리오 알림 컴포넌트 배치.
404 Not Found 요청한 자원(특정 Statement ID, State, Profile 등)이 존재하지 않는 경우. 조회(GET) 데이터 기획 시, 가상의 Mockup ID를 매핑하고 데이터 없을 시 UI 예외 처리 설계 지원.
409 Conflict 이미 존재하는 Statement ID에 대해 다른 데이터를 다시 PUT/POST 하여 데이터 무결성/불변성을 침해하는 경우. UUIDv4 자동 생성 기능을 기본 활성화하여 ID 충돌을 원천 차단하고 불변성(Immutability) 규칙 안내 문구 노출.
412 Precondition Failed State API 등에서 제공된 ETag 등 조건부 헤더 값이 서버 상태와 일치하지 않는 경우. 동적 데이터 업데이트 기획 시 동시성 제어를 위한 ETag 처리 가이드라인 및 필수 헤더 매핑 UI 구현.
413 Request Entity Too Large LRS가 허용하는 최대 크기(또는 첨부파일 크기)를 초과하는 Statement를 전송한 경우. 첨부파일(Attachments) 추가 시 용량 제한(예: 5MB)을 미리 설정하고 초과 시 경고 팝업 발생.

3. xAPI 핵심 데이터 구조 및 유효성 검증 규칙

기획 툴에서 드래그 앤 드롭으로 조합할 핵심 컴포넌트들의 데이터 규격 규칙입니다. 기획자가 필수값을 누락할 경우 붉은색(Red Border) 시각 효과와 함께 가이드를 제공해야 합니다.

3.1 Statement 구조 및 필수 항목 규칙

1. Actor (행위자) 필수
  • objectType: "Agent" 또는 "Group"
  • 역기능 식별자(IFI) 중 정확히 1개만 필수 포함:
    • mbox: 반드시 mailto:로 시작하는 이메일 형태
    • mbox_sha1sum: 이메일 SHA-1 해시값
    • openid: 유효한 OpenID IRI
    • account: homePage(IRI)와 name의 쌍
2. Verb (동사) 필수
  • id: 의미를 정의하는 유효한 IRI(URL/URI) 구조 형태여야 함.
  • display: 기획자 가독성을 위한 언어 맵(Language Map, 예: {"ko-KR": "완료했다"}).
3. Object (객체) 필수
  • id: 액티비티를 식별하는 고유 IRI.
  • definition: 타입, 명칭, 설명, 확장(Extensions) 정의.

3.2 부가 데이터 항목 규칙

  • Result (결과) 선택: 점수(Score: scaled, raw, min, max), 성공 여부(success), 완료 여부(completion), 소요시간(duration - ISO 8601 Duration 포맷 규칙 준수 필요) 포함 가능.
  • Context (맥락) 선택: 등록 ID(registration UUID), 상위/그룹 액티비티 관계(parent, grouping), 강사(instructor) 등 메타데이터 바인딩.
  • Timestamp (발생 일시) 권장: ISO 8601 시간 포맷(예: 2026-05-27T12:43:00Z) 준수 필수. 누락 시 LRS 저장 시점 자동 생성되나 기획 권장.

4. 데이터 기획자를 위한 사이트(기획 툴) UI/UX 및 기능 설계

제안하신 핵심 아이디어를 구체화하여, 데이터 기획자가 코딩 없이 드래그 앤 드롭 및 시각적 피드백을 통해 완벽한 xAPI 로깅 시나리오를 만들 수 있는 아키텍처 가이드라인입니다.

4.2 실시간 데이터 유효성 검사 (Validation) 로직 흐름

  1. 컴포넌트 배치 단계: 기획자가 좌측 패널에서 Actor, Verb, Object 컴포넌트를 캔버스로 드래그하면, 메인 빌더 내부에 빈 인스턴스 템플릿이 JSON 형태로 자동 생성됩니다.
  2. 실시간 포맷 체커 (Red-Highlight Trigger):
    • 필수값 체크: Actor, Verb, Object 가 비어있거나 하위 세부 입력란이 채워지지 않은 경우, 해당 컴포넌트의 컨테이너 테두리를 border: 2px solid #ef4444; 및 background-color: #fef2f2;로 변경하여 시각적 경고 유도.
    • 데이터 타입 유효성 검사:
      • mbox 입력 칸에 mailto: 접두사 없이 일반 이메일 주소(예: user@test.com)만 입력하면 정규식 필터링을 통해 빨간색 경고 메시지 표시 및 자동으로 mailto: 보정 처리 버튼 제공.
      • Verb ID 및 Activity ID에 유효하지 않은 URL 주소 포맷 입력 시 경고 처리.
  3. 프로파일(Profile) 및 사전 정의 딕셔너리 연동: 기획자가 텍스트를 직접 치지 않고, 미리 정의된 ADL 표준동사 딕셔너리(예: initialized, completed, passed, answered) 목록에서 드롭다운으로 선택할 수 있도록 구성하여 오탈자로 인한 규격 위반 사전 차단.
💡 데이터 기획자용 사이트 핵심 셀링 포인트 (Value Proposition)
본 사이트는 단순한 JSON 에디터가 아닌, "LRS 시뮬레이션 기반 데이터 기획 샌드박스"입니다. 기획 단계에서 완성된 스키마 흐름을 시각적으로 확인하고 한 클릭으로 설계된 JSON 명세와 데이터 설계 기술서 문서를 자동 수출(Export)할 수 있어, 개발자와 기획자 간의 커문니케이션 비용을 90% 이상 절감할 수 있습니다.
